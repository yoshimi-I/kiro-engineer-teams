
# E2Eバグハント — Playwrightで全ページを巡回しバグをissue化するループ

Playwrightでアプリの全ページ・全機能を操作し、バグ・UI崩れ・コンソールエラーを発見次第GitHub issueを作成する。

## 検出対象

| カテゴリ | issueラベル | 判定基準 |
|---------|------------|---------|
| クラッシュ・エラー | `bug` | ページ遷移失敗、コンソールエラー、未処理例外 |
| UI崩れ | `bug, frontend` | 要素重なり、はみ出し、切れ、レスポンシブ崩れ |
| 機能不動作 | `bug` | ボタン無反応、WebSocket切断、状態不整合 |
| UX問題 | `enhancement` | 操作が分かりにくい、導線不明 |
| アクセシビリティ | `enhancement` | コントラスト不足、タッチ領域小、aria-label欠如 |

## 開始前の準備（必須）

1. ルーティング定義を読む（App.tsx, router.ts等）→ 全ルートを把握
2. 既存E2Eテストを読む（e2e/）→ テストパターン・ヘルパーを把握
3. Playwright設定を読む（playwright.config.ts）→ baseURL, webServer, timeout確認
4. 既存issueを取得 → 重複防止

## 巡回シナリオ（サイクルごとにローテーション）

1. **全ページ表示チェック**: 各ページに遷移、クラッシュしないこと・主要要素表示を確認
2. **主要ユーザーフロー**: アプリの主要操作フローを一通り実行
3. **フォーム・入力系**: 正常入力・空入力・境界値入力、バリデーション確認
4. **レスポンシブ検証**: 全ページをモバイル幅（375px）で表示
5. **エッジケース**: ブラウザバック、存在しないURL、連打
6. **アクセシビリティ**: キーボード操作、フォーカス管理、aria-label

## 1サイクルの処理手順

1. `e2e/` に一時テストファイルを動的生成（既存テストのパターンに従う）
2. `npx playwright test e2e/<テストファイル> --reporter=json` で実行
3. 失敗したテストごとに:
   - スクリーンショット確認、コンソールエラー収集
   - 既存issueと重複チェック
   - 重複なし → `gh issue create`
4. 一時テストファイルを削除
5. 次のサイクルへ

## テスト生成のルール

- 既存の `e2e/*.spec.ts` のパターンに従う
- 全テストでコンソールエラーを収集する
- 各ステップでスクリーンショットを撮影する

## ループ停止条件

| 条件 | 動作 |
|------|------|
| 全6サイクル完了 | 報告して停止 |
| ユーザーが「止めて」と言った | 即座に停止 |
| アプリが起動しない | エラー報告して停止 |

## よくあるミス

- スクリーンショットを撮らない → 各ステップで必ず撮影
- コンソールエラーを無視 → 必ず収集
- 1つのissueに複数バグをまとめる → 1バグ = 1 issue
- 既存issueと重複チェックしない → 必ず確認
- 一時テストファイルを残す → 必ず削除
