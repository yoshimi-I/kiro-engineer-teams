
# コード巡回・問題発見ループ

コードベースを体系的に巡回し、潜在的なバグ・設計問題・品質劣化を発見 → issue作成を繰り返す。

## 巡回カテゴリ（サイクルごとにローテーション）

| # | 視点 | 調査内容 | issueラベル |
|---|------|---------|------------|
| 1 | セキュリティ | injection, 認証バイパス, シークレット漏洩, XSS, 入力未検証 | `security` |
| 2 | ロジック不整合 | エッジケース, null/undefined未処理, off-by-one, 型不一致 | `bug` |
| 3 | アーキテクチャ違反 | レイヤー越境, 循環依存, 責務混在 | `code-quality` |
| 4 | パフォーマンス | N+1クエリ, 不要な再レンダリング, O(n²), メモリリーク候補 | `performance` |
| 5 | エラー処理 | 未捕捉例外, 握りつぶし, リソースリーク, 不十分なログ | `bug` |
| 6 | フロント↔バック整合性 | 型定義の乖離, API仕様とコードの不一致, WebSocketメッセージ型不整合 | `bug` |
| 7 | 未使用・重複コード | デッドコード, コピペ実装, 未使用import/export | `code-quality` |
| 8 | テストカバレッジ | テスト未実装の重要ロジック, エッジケース未テスト | `testing` |

## 1サイクルの処理

1. サイクル番号 `N` を `N % 8` で巡回カテゴリを決定
2. 前回の巡回記録（`issue/patrol-log.md`）を読み、調査済みファイルを把握
3. 未調査のディレクトリ/ファイルを選択（`backend/app` → `frontend/src` → `infra/` をローテーション）
4. 対象ファイルを最低10ファイル読み、該当カテゴリの視点で問題を探す
5. 問題発見時: 既存issueと重複チェック → `gh issue create` で1問題=1issue作成
6. 巡回記録を更新
7. 次のサイクルへ（2分間隔）

## issue作成基準

| 基準 | 閾値 |
|------|------|
| 信頼度 | 80以上のみ作成（推測・憶測でissueを作らない） |
| 再現性 | コードから明確に問題が読み取れること |
| 影響度 | ユーザー影響 or データ不整合 or セキュリティリスクがあること |
